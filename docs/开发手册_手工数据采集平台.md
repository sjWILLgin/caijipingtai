# 手工数据采集平台 开发手册

## 1. 目标与读者
本文档面向开发同学，帮助快速理解架构、代码结构、接口、数据模型和二次开发方式。

## 2. 项目结构
- backend：Node.js + Express + TypeScript
- frontend：React + Vite + Ant Design

关键后端目录：
- src/index.ts：服务入口
- src/routes：API 路由层
- src/services：业务处理层
- src/db.ts：数据库连接池

关键前端目录：
- src/services/api.ts：接口封装
- src/pages：页面组件
- src/components：通用组件

## 3. 环境配置
后端 .env 关键项：
- DB_HOST
- DB_PORT
- DB_USER
- DB_PASSWORD
- DB_NAME
- TARGET_DB_NAME
- META_DB_NAME
- PORT
- UPLOAD_DIR
- IMPORT_CHUNK_SIZE
- IMPORT_MAX_ROWS
- IMPORT_MAX_TARGET_ROWS
- ROLLBACK_SNAPSHOT_MAX_ROWS
- JOB_QUEUE_CONCURRENCY

生产约束：
- `DB_HOST` 必须配置为阿里云 RDS 地址，当前后端会阻止 `localhost`、`127.0.0.1` 等本地数据库地址启动。
- `DB_NAME` 存放平台系统表与流程元数据。
- `TARGET_DB_NAME` 存放业务目标表，校验、映射、入库、表清单读取均以该库为准。
- `META_DB_NAME` 存放回滚快照与辅助元数据。

本地默认：
- 后端：3001
- 前端：5174

## 4. 数据库模型
核心表：
- import_plan
- import_task
- import_file
- sheet_mapping
- field_mapping
- validate_rule
- validate_error
- import_batch
- audit_log
- async_job
- 目标业务表（如 hp_sfa_business_group）

分层建议：
- 元数据库（`DB_NAME`）：系统与流程元数据（用户、权限、审批、任务、日志等）
- 目标数据库（`TARGET_DB_NAME`）：业务目标表与同步结果数据
- 快照库（`META_DB_NAME`）：覆盖类入库回滚快照（通常与 `DB_NAME` 一致）

## 5. 核心业务流程（开发视角）
1. 创建任务：`POST /api/import-tasks`
2. 上传并解析：`POST /api/import-files/upload` -> `parseService.parseFile`
3. 保存 Sheet 配置：`PUT /api/import-tasks/:taskId/sheet-mappings`
4. 保存映射：`PUT /api/import-tasks/:taskId/mappings`
5. 执行校验：`POST /api/validation/:taskId/run` -> `validationService.runValidation`
6. 提交入库：`POST /api/commit/:taskId` -> `commitService.commitData`
7. 回滚：`POST /api/commit/batches/:batchId/rollback`

## 6. 重点实现说明

## 6.0 审批流引擎（新增）
核心服务：`services/approvalFlowService.ts`
- 模板能力：创建、编辑、发布、查询详情。
- 节点策略：
  - `SERIAL`：串行节点，逐级推进
  - `OR_SIGN`：或签，任一满足条件即可通过
  - `AND_SIGN`：会签，需满足通过阈值
- 审批人模型：
  - `USER`：指定用户
  - `ROLE`：按角色
  - `DOMAIN_ADMIN`：按业务域管理员
- 阈值规则：
  - 通过阈值：`MIN_PASS_COUNT`
  - 驳回阈值：`THRESHOLD_REJECT`
- 与提交联动：`routes/commit.ts` 在命中审批条件时创建/推进流程实例，未通过前阻断最终入库。

审批命中口径（当前生效）：
- 只按目标表命中启用模板（`enabled=1` 且 `target_tables_json` 包含目标表）
- 不再按业务域参与命中优先级排序，避免同表跨域参数判定不一致

## 6.1 自动映射
路由：`routes/mappings.ts`
- `mapping_mode=SAME_NAME`：按字段名匹配。
- `mapping_mode=ORDER`：按序号匹配。

## 6.2 映射后强制重校验
路由：`routes/tasks.ts` 的 mappings 保存逻辑
- 清空 `validate_error`
- 重置任务统计字段
- 状态切回需要校验阶段

## 6.3 类型校验
服务：`services/validationService.ts`
- 对映射字段根据目标列 `DATA_TYPE` 做校验：
  - 数值类型：必须可转 number
  - 日期类型：必须可解析
- 错误类型：`TYPE_MISMATCH`
- 目标列元数据来源：`TARGET_DB_NAME`（避免误读元数据库）。

## 6.4 入库与回滚
路由：`routes/commit.ts`
- 入库状态门禁：READY/VALIDATE_SUCCESS。
- 入库通过异步作业执行（`async_job` + `jobQueue`）。
- 回滚策略：
  - APPEND：按 batch_id 删除。
  - FULL_OVERWRITE / PARTITION_OVERWRITE / UPSERT：优先从元仓 `data_collection_meta` 快照恢复提交前状态。
  - 无快照时回退到兼容策略。
- 支持“已失效批次”的补偿清理。
- 目标表 schema 查询与 DML 全部按 `TARGET_DB_NAME` 执行。
- 覆盖类模式的回滚快照写入 `META_DB_NAME`，与目标库解耦。

## 6.5 目标字段发现与表清单隔离（新增）
- 路由：`routes/mappings.ts`
  - 目标字段发现从 `TARGET_DB_NAME` 的 `INFORMATION_SCHEMA` 获取。
- 路由：`routes/tables.ts`
  - 表清单接口直接读取数据库真实表，不依赖 `import_plan` 历史数据。
  - 初始化后即使导入方案为空，手工数据表列表仍可正常展示。
  - 系统前缀与核心元数据表被过滤，避免用户误操作元表。

## 6.6 大数据防爆策略（新增）
目标：回答“大数据会不会把服务和数据库搞爆”。

1. 内存保护
- 上传 hash 使用流式计算，避免整文件读取到内存。
- 解析、校验、入库均采用分块处理（`IMPORT_CHUNK_SIZE`）。
- 总处理行数上限（`IMPORT_MAX_ROWS`），超限即阻断并提示分批导入。

2. 并发与异步保护
- PARSE / VALIDATE / COMMIT 全部通过 `async_job` 队列执行。
- 默认串行并发（`JOB_QUEUE_CONCURRENCY=1`），防止数据库瞬时写入冲击。

3. 数据库容量保护
- 目标表行数上限：`IMPORT_MAX_TARGET_ROWS`。
- 对 APPEND / UPSERT / PARTITION_OVERWRITE 先做预计行数校验，超过上限直接拒绝入库。

4. 回滚快照容量保护
- 元仓快照行数上限：`ROLLBACK_SNAPSHOT_MAX_ROWS`。
- 超限则阻断覆盖类入库，避免 `data_collection_meta` 爆涨。

5. 排障口径
- 如果失败信息为 `ROW_LIMIT_EXCEEDED`：单任务行数过大。
- 如果失败信息为 `TARGET_TABLE_ROW_LIMIT_EXCEEDED`：目标表增长风险。
- 如果失败信息为 `SNAPSHOT_TOO_LARGE`：回滚快照过大。
- 处理建议：拆批导入、归档历史数据、调整限额参数后重试。

## 6.7 任务删除（单删/批删）
路由：`routes/tasks.ts`
- `DELETE /api/import-tasks/:taskId`
- `POST /api/import-tasks/batch-delete`
- 级联清理任务关联记录 + 目标表数据 + 文件记录。

## 6.8 权限模型与导入链路默认权限
核心文件：
- `services/permissionMatrix.ts`：权限键、角色默认权限、默认权限合并方法。
- `middleware/permissionMap.ts`：按接口路径 + HTTP 方法映射权限键。
- `middleware/auth.ts`：角色与显式权限校验。
- `services/authInit.ts`：系统启动时初始化 root、角色、默认权限。

当前口径：
- `super_admin` 默认拥有全部权限，且仅 root 可担任。
- `analyst` 和 `domain_admin` 在运行时会合并角色默认权限与用户额外勾选权限。
- 导入链路基础权限属于角色默认权限，包括 `plan.manage`、`task.create`、`task.delete`、`file.upload`、`mapping.edit`、`validation.run`、`commit.execute`、`commit.rollback` 等。
- 用户权限页的勾选用于追加扩展权限，不用于取消角色基础导入权限。
- 页面可见性仍由前端按 `role_key` 控制，因此分析师看到的菜单少于超管，但接口执行导入链路不会被 `plan.manage` 阻断。

## 7. API 约定
统一响应结构：
- success: boolean
- message: string
- data: any

错误处理：
- 业务错误返回 4xx
- 系统错误返回 500

## 8. 安全要点
- 动态表名必须正则白名单：`^[a-zA-Z0-9_]+$`
- 上传文件类型与大小限制
- 不信任前端传入状态，服务端强校验状态机

## 9. 日志与排障
推荐排查顺序：
1. 任务状态（import_task）
2. 批次记录（import_batch）
3. 校验错误（validate_error）
4. 审计日志（audit_log）
5. 目标表数据（按 batch_id / task_id）

常用 SQL：
- 查任务：
  `SELECT * FROM import_task WHERE task_id=?;`
- 查批次：
  `SELECT * FROM import_batch WHERE task_id=? ORDER BY created_at DESC;`
- 查错误：
  `SELECT * FROM validate_error WHERE task_id=? ORDER BY sheet_name,row_no;`

## 11. 发布与回归建议
发布前最小回归：
1. 新建方案 + 导出模板
2. 上传/解析/Sheet配置/映射/校验/入库全链路
3. 回滚（FULL_OVERWRITE 与 APPEND）
4. 任务单删与批删
5. 错误导出与日志可视化

## 12. 2026-05-25 开发增量

### 12.1 Dashboard 分层接口
- `GET /api/dashboard/stats`：保留首页核心大盘指标。
- `GET /api/dashboard/health`：新增数据健康度明细（域级表数、贡献度、容量、任务质量、审批积压、高容量表 TopN）。
- 两个接口均纳入 `dashboard.view` 权限映射。

### 12.2 域隔离实现
- `routes/plans.ts`：list/get/create/update/enable/disable/delete 全量加入域访问断言。
- `routes/tables.ts`：`GET /api/tables` 支持 `domain` 参数，非超管仅可见自身域目标表。
- `P02PlanForm` 前端联动：切换业务域后重新拉取目标表并清理跨域旧值。

### 12.3 手工表治理增强
- 建表申请接口增加重复拦截。
- 建表进度状态接口与前端进度卡片联动（pending 自动轮询）。
- 删表接口新增角色硬校验、删除原因、删后 existence 校验与双留痕。

### 12.4 安全与配置
- `.env.example` 使用占位主机和示例账号，不包含真实连接信息。
- root 初始密码改为环境变量 `ROOT_INIT_PASSWORD` 控制。
- 生产部署必须在上线前显式设置 `ROOT_INIT_PASSWORD` 并完成首次改密。
