# 手工数据采集平台部署指南（百万级导入）

## 1. 文档目标
本文面向运维人员，目标是可以直接完成打包、上传、部署、启动、验证与回滚。

部署形态：
- 前端：React + Vite 构建为静态文件，由 Nginx 托管。
- 后端：Node.js + Express + TypeScript，构建为 `backend/dist`，由 PM2 常驻运行。
- 数据库：生产环境使用阿里云 RDS MySQL，当前后端启动时会阻止 `localhost`、`127.0.0.1` 等本地数据库地址。

生产访问链路：

```text
用户浏览器 -> Nginx 80/443 -> 前端静态文件
用户浏览器 -> Nginx /api、/uploads 反向代理 -> 后端 127.0.0.1:3001
后端 -> 阿里云 RDS MySQL
```

## 3. 推荐规格（默认百万行上传）

### 3.1 测试环境
- 应用机：4 vCPU / 8 GB / 100 GB SSD
- 数据库：RDS 或测试 MySQL

### 3.2 生产推荐
- 应用机：8 vCPU / 16 GB / 200 GB SSD
- 数据库：8 vCPU / 32 GB / 500 GB SSD 以上 RDS
- 网络：应用机与 RDS 同地域、同 VPC，优先内网访问

### 3.3 并发导入增强
- 应用机：16 vCPU / 32 GB
- 数据库：16 vCPU / 64 GB 以上 RDS
- 大批次导入安排在业务低峰

## 4. 目录规划

建议统一使用以下目录：

```text
/opt/data-collection                 应用部署目录
/opt/data-collection/backend         后端目录
/opt/data-collection/frontend        前端目录
/var/www/data-collection             Nginx 前端静态站点目录
/var/log/data-collection             应用日志目录
/data/backup/data-collection         备份目录
```

说明：
- 当前代码上传文件固定写入 `backend/uploads`，请确保该目录可写。
- Nginx 的 `/uploads/` 会反向代理到后端 `/uploads/`。

## 5. 数据库准备

### 5.1 数据库分层

生产建议在同一个阿里云 RDS MySQL 实例创建三个库：

| 环境变量 | 推荐库名 | 用途 |
|---|---|---|
| `DB_NAME` | `data_collection_platform` | 系统表、任务、权限、审批、日志等元数据 |
| `TARGET_DB_NAME` | `data_collection_target` | 业务目标表与最终导入数据 |
| `META_DB_NAME` | `data_collection_meta` | 回滚快照与辅助元数据 |

### 5.2 创建数据库与账号

使用 RDS 管理控制台或具有权限的 MySQL 客户端执行：

```sql
CREATE DATABASE IF NOT EXISTS data_collection_platform DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE DATABASE IF NOT EXISTS data_collection_target DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE DATABASE IF NOT EXISTS data_collection_meta DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE USER IF NOT EXISTS 'data_collection_app'@'%' IDENTIFIED BY '请替换为强密码';
GRANT ALL PRIVILEGES ON data_collection_platform.* TO 'data_collection_app'@'%';
GRANT ALL PRIVILEGES ON data_collection_target.* TO 'data_collection_app'@'%';
GRANT ALL PRIVILEGES ON data_collection_meta.* TO 'data_collection_app'@'%';
FLUSH PRIVILEGES;
```

RDS 安全要求：
- RDS 禁止公网访问，应用优先使用 RDS 内网地址。
- RDS 白名单加入应用机 ECS 内网 IP 或安全组。
- 应用使用独立账号，不建议使用 RDS root 账号。

### 5.3 业务目标表

导入前需确保 `TARGET_DB_NAME` 中已存在目标业务表。平台会根据目标表结构导出模板、字段映射、校验并写入数据。

业务目标表建议：
- 使用 `utf8mb4` 字符集。
- 表名与字段名使用字母、数字、下划线。
- 按业务查询场景建立必要索引。
- 百万级导入目标表建议预留足够磁盘和 undo/redo 空间。

## 7. 打包

### 7.1 在打包机或服务器构建

进入项目根目录：

```bash
cd /path/to/采集平台
```

安装依赖并构建：

```bash
cd backend
npm ci
npm run build

cd ../frontend
npm ci
npm run build
```

构建产物：
- 后端：`backend/dist`
- 前端：`frontend/dist`

### 7.2 生成部署包

回到项目根目录：

```bash
cd /path/to/采集平台

tar \
  --exclude='backend/node_modules' \
  --exclude='frontend/node_modules' \
  --exclude='.git' \
  --exclude='backend/.env' \
  --exclude='backend/uploads' \
  -czf data-collection-release-$(date +%Y%m%d%H%M%S).tar.gz \
  backend/package.json \
  backend/package-lock.json \
  backend/dist \
  backend/.env.example \
  frontend/dist \
  docs \
  README.md
```

说明：
- 部署包不包含 `.env`，生产环境变量在服务器单独维护。
- 部署包不包含 `uploads`，避免覆盖线上上传文件。
- 后端运行仍需安装生产依赖，服务器执行 `npm ci --omit=dev`。

## 10. 配置 Nginx

创建配置文件：

```bash
sudo vi /etc/nginx/conf.d/data-collection.conf
```

写入：

```nginx
server {
  listen 80;
  server_name your-domain.com;

  client_max_body_size 200m;

  root /var/www/data-collection;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  location /api/ {
    proxy_pass http://127.0.0.1:3001/api/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_connect_timeout 60s;
    proxy_send_timeout 300s;
    proxy_read_timeout 300s;
  }

  location /uploads/ {
    proxy_pass http://127.0.0.1:3001/uploads/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_read_timeout 300s;
  }
}
```

检查并重载：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

HTTPS 建议由公司统一证书或 SLB/Nginx 证书配置实现。生产环境建议只开放 80/443，对外不要开放 3001。

## 12. 百万级上传参数建议

### 12.1 后端参数

推荐：

```env
IMPORT_CHUNK_SIZE=2000
IMPORT_MAX_ROWS=1200000
IMPORT_MAX_TARGET_ROWS=10000000
ROLLBACK_SNAPSHOT_MAX_ROWS=10000000
JOB_QUEUE_CONCURRENCY=1
```

说明：
- `IMPORT_CHUNK_SIZE` 越大，单批处理吞吐越高，但内存和数据库压力也更大。
- `IMPORT_MAX_ROWS` 是单次解析、校验、入库行数上限。
- `JOB_QUEUE_CONCURRENCY=1` 更稳，避免多个百万级导入同时压垮数据库。

### 12.2 Nginx 上传限制

建议：

```nginx
client_max_body_size 200m;
proxy_read_timeout 300s;
```

### 12.3 MySQL / RDS 建议

- `max_allowed_packet`：64M 或更高。
- `innodb_buffer_pool_size`：RDS 按规格配置，建议使用高内存规格。
- 开启慢查询日志，`long_query_time=1`。
- 大批量导入前确认目标库磁盘余量。

## 14. 回滚

### 14.1 前端回滚

将备份的前端目录恢复到 `/var/www/data-collection`：

```bash
rm -rf /var/www/data-collection/*
cp -r /data/backup/data-collection/备份时间/frontend/* /var/www/data-collection/
sudo nginx -t && sudo systemctl reload nginx
```

### 14.2 后端回滚

如果保留了上一个部署包，可重新解压旧包并重启：

```bash
cd /opt/data-collection
tar -xzf /tmp/data-collection-release-旧版本.tar.gz

cd /opt/data-collection/backend
npm ci --omit=dev
cp /data/backup/data-collection/备份时间/backend.env .env
pm2 restart data-collection-backend
```

数据库结构变更回滚需结合 RDS 快照或备份策略执行，生产回滚前需先确认数据影响。

## 15. 常见故障排查

### 15.1 页面打不开

```bash
curl -I http://your-domain.com/
sudo nginx -t
sudo systemctl status nginx
```

如果 Nginx 正常，再确认前端文件是否存在：

```bash
ls -lah /var/www/data-collection
```

### 15.2 接口 502

通常是后端没启动或端口不通：

```bash
lsof -nP -iTCP:3001 -sTCP:LISTEN
pm2 status
pm2 logs data-collection-backend --lines 100
curl -i http://127.0.0.1:3001/api/health
```

### 15.3 后端启动失败：DB_HOST 非阿里云RDS地址

当前代码要求生产 `DB_HOST` 必须包含 `aliyuncs.com`，不能使用：
- `localhost`
- `127.0.0.1`
- 非阿里云 RDS 地址

修复：

```bash
vi /opt/data-collection/backend/.env
pm2 restart data-collection-backend
```

将 `DB_HOST` 改为 RDS 内网地址，例如：

```env
DB_HOST=rm-xxxxxxxx.mysql.rds.aliyuncs.com
```

### 15.4 登录失败或 401

检查：
- 浏览器是否访问同一个域名，避免 token 存在另一个域。
- 后端 `JWT_SECRET` 是否在重启或多实例之间保持一致。
- 系统时间是否异常。

### 15.5 上传失败或 413

检查 Nginx：

```nginx
client_max_body_size 200m;
```

修改后：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### 15.6 无权限执行该操作：plan.manage

当前版本已将 analyst 和 domain_admin 的导入链路基础执行权限补齐。若仍出现该错误：

```bash
pm2 restart data-collection-backend
```

然后重新登录，让前端刷新用户权限信息。

### 15.7 查看日志

```bash
pm2 logs data-collection-backend --lines 200
tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log
```

## 16. 安全建议

- 生产必须使用 HTTPS。
- `JWT_SECRET` 不可使用默认值，应使用高强度随机串。
- 只开放 80/443，后端 3001 仅本机或内网可访问。
- RDS 只允许应用机内网 IP 访问。
- 生产 `.env` 不进入部署包、不提交代码仓库。
- root 初始密码上线后立即修改。
- 定期备份 `DB_NAME`、`TARGET_DB_NAME`、`META_DB_NAME`。
